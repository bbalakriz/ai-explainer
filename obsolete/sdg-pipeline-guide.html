Synthetic Data Generation Pipeline

A pipeline that generates Thai-language training data for AIS telecom intent classification. Reads seed examples, builds structured LLM prompts, and produces validated multi-turn chat datasets across 16 intent categories.

Getting Started

Quick Start

Copy any command below to start generating. The standalone mode requires only litellm and pyyaml.

Default python run.py --model "gemini/gemma-4-31b-it" --api-key $GOOGLE_API_KEY --num-examples 1 --max-seeds 5
Flow mode python run.py --model "gemini/gemma-4-31b-it" --api-key $GOOGLE_API_KEY --num-examples 1 --max-seeds 5 --mode flow
OpenAI python run.py --mode standalone --model openai/gpt-4o --output ../generated
Local LLM python run.py --mode standalone --model ollama/gemma2 --api-base http://localhost:11434
Single intent python run.py --intents MOBILE_BILLING_CHECK_DUE_DATE --num-examples 10
File Layout

Project Structure

All paths are relative to the repository root. The pipeline reads from samples/ and writes to generated/.

sdg/ samples/ └─ training_dataset.json ← 16 intents, ~400 seed examples pipeline/ ├─ run.py ← Main pipeline script (entrypoint) ├─ run_sdg_pipeline.ipynb ← Notebook version ├─ intent_descriptions.json ← 16 intent descriptions (shared) ├─ flow.yaml ← sdg_hub flow definition (3 blocks) └─ prompts/ └─ generate_telco_chat.yaml ← LLM prompt template generated/ ← Output directory └─ {INTENT_NAME}.json ← One file per intent
Execution Flow

Step-by-Step Pipeline

Click any step to expand its details. The pipeline flows top-to-bottom, with animated connectors showing the data path.

1
CLI Argument Parsing
main() → argparse.ArgumentParser
▼

The main() function defines 8 CLI arguments via argparse. The two most important are --mode (which backend to use) and --model (which LLM to call). All arguments have sensible defaults so the pipeline can run with zero configuration.

Shell
python run.py \
  --mode standalone  \  # or "flow"
  --model anthropic/claude-sonnet-4-20250514 \
  --num-examples 20  \  # per intent
  --max-seeds 5     \  # few-shot examples in prompt
  --intents MOBILE_BILLING_CHECK_DUE_DATE  # optional filter
2
Load Intent Descriptions
Module-level → intent_descriptions.json
▼

At module import time (before main() runs), the script loads intent_descriptions.json into the INTENT_DESCRIPTIONS dictionary. This file is shared with the notebook so both entrypoints always have the same descriptions. Each description is a plain-English sentence explaining what the customer is trying to do.

Python — run.py:34-36
_INTENT_DESC_PATH = Path(__file__).parent / "intent_descriptions.json"
with open(_INTENT_DESC_PATH, encoding="utf-8") as _f:
    INTENT_DESCRIPTIONS: dict[str, str] = json.load(_f)

These descriptions are injected into the LLM prompt (step 5) so the model understands the semantic meaning of each intent beyond just its name.

3
Load Seed Data
load_seed_data() → training_dataset.json
▼

Reads samples/training_dataset.json — a single JSON file containing ~400 human-crafted examples across 16 intents. The function groups them by intent key into a dictionary.

Example Seed Record
user_message"แล้วสรุปต้องจ่ายวันไหนครับ"
intent"MOBILE_BILLING_CHECK_DUE_DATE"
session_history[ {role: "user", content: "..."}, {role: "assistant", content: "..."} ... ]

Each seed example has three fields: the user's Thai message, the intent label, and an optional conversation history (up to 5 prior messages).

4
Select Seed Examples
select_seed_examples() → diversity sampling
▼

For each intent, the pipeline selects up to max_seeds (default 5) examples to include as few-shot demonstrations in the prompt. The selection algorithm ensures diversity by grouping examples by their session_history length and round-robin sampling across buckets.

Diversity Bucketing Strategy

0 msgs
1 msg
2 msgs
3 msgs
4 msgs
5 msgs

One example is taken from each bucket in round-robin order until 5 are selected. Any history longer than 5 messages is truncated to the last 5 (sliding window), matching the real system's constraint.

5
Build Prompt
build_prompt() → generate_telco_chat.yaml
▼

Loads the YAML prompt template and performs variable substitution. The result is a 2-message chat array (system + user) sent to the LLM. This is the core of the pipeline — the prompt quality directly determines output quality.

Template Variables

{{ intent }} Intent name
{{ intent_description }} From JSON
{{ seed_examples }} Few-shot JSON
{{ num_examples }} Target count
{{ num_seed }} Seed count

See the full Prompt Anatomy section below for the complete system and user message breakdown.

6
LLM Generation
litellm.completion() or sdg_hub.Flow
▼

The prompt is sent to the chosen LLM. The execution path diverges based on the selected mode:

STANDALONE MODE
Calls litellm.completion() directly for each intent, one at a time. Parameters: max_tokens=16384, temperature=0.8, timeout=1800s.
FLOW MODE
Passes all intents through a 3-block sdg_hub.Flow pipeline with up to 5 concurrent LLM calls via async_mode.

The LLM returns a JSON array of generated training examples. See Modes Comparison for a detailed breakdown.

7
Parse & Validate
parse_json_output() + validate_examples()
▼

Parsing: Strips markdown code fences if present, locates the first [ and last ], and attempts json.loads(). Falls back gracefully on failure.

Validation Checks

✓Must be a dict (non-dict entries are dropped)
✓Must contain user_message field
✓Forces intent to the correct value (overrides LLM output)
✓Validates session_history is a list of {role, content}
✓Only allows role: "user" or "assistant"
✓Enforces sliding window: max 5 messages in history
✓Outputs only 3 fields: user_message, intent, session_history
8
Save & Summarize
json.dump() → generated/{INTENT}.json
▼

Each intent's validated examples are written to their own JSON file in the generated/ directory. After all intents are processed, a summary is printed showing per-file counts and the grand total.

Prompt Engineering

Prompt Anatomy

The LLM prompt is a 2-message chat template loaded from generate_telco_chat.yaml. Each section serves a specific purpose in controlling output quality.

◆ System Message

Establishes the LLM's role as an expert synthetic data generator for Thai telecom chat data. Contains five critical rule categories:

◆
Language & Format
Generate Thai with natural English loanwords (roaming, package, data). Mask phone numbers as 091234XXXX and OTPs as XXXXXX.
◆
Sliding Window
session_history must contain at most 5 messages. This matches the real production system's context window.
◆
Typos & Informal Language
8-10 of 20 examples must include realistic Thai typos, misspellings, emoji, slang like "555", "ค้าบ", "ป่าว", doubled vowels.
◆
Sentence Diversity
Vary verb choice, sentence form (question/statement/imperative/fragment), directness level, and message length. No repeated phrasing.
◆
Intent Changes in History
5-6 of 20 examples must have session_history about a different topic before the user switches to the target intent. Simulates real topic-switching.
◆
Assistant Style
Polite Thai customer service tone with ค่ะ/คะ particles. Include [Options: {...}] in some responses for interactive choices.
◆ User Message

Requests generation of exactly N examples for a specific intent. Includes the intent description, seed examples as JSON, and 11 detailed requirements:

1 message

2 messages

3 messages

Additional Requirements

▸5-6 examples with topic switch in history
▸8-10 examples with Thai typos/slang/emoji
▸Mix formal/informal, male/female particles
▸Some with phone numbers as the message
▸Some confirming assistant-presented options
▸2-3 examples in English (non-native style)

Configuration

CLI Reference

All arguments are optional with sensible defaults. The minimum viable command is just python run.py.

Argument Default Description
--mode standalone Execution backend: standalone (litellm) or flow (sdg_hub)
--model anthropic/claude-sonnet-4-20250514 LLM model identifier in litellm format (provider/model-name)
--api-base None Base URL for self-hosted models (e.g., Ollama at localhost:11434)
--api-key None API key. Can also use env vars: ANTHROPIC_API_KEY, OPENAI_API_KEY
--output ../generated Output directory for generated JSON files
--num-examples 20 Number of synthetic examples to generate per intent
--max-seeds 5 Maximum seed examples to include as few-shot demonstrations
--intents all Space-separated list of specific intents to generate
Model Support

Compatible Models

Any model supported by litellm works. These are the recommended and tested options from flow.yaml.

anthropic/claude-sonnet-4-20250514 default
openai/gpt-4o
meta-llama/Llama-3.3-70B-Instruct
ollama/gemma2 local

For local models, use --api-base http://localhost:11434. The 1800s timeout accommodates slower inference on consumer hardware.
