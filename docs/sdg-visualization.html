Architecture

Pipeline Flow

Five stages transform human-curated seed data into hundreds of validated training examples, maintaining format consistency and linguistic realism.

1

Seed Data Loading

load_seed_data() → select_seed_examples()
  • Reads 10 JSON files with existing human-curated examples (16–26 per intent)
  • Each file maps to one intent — e.g. MOBILE_BILLING_CHECK_DUE_DATE.json
  • Smart seed selection: round-robin across session_history length buckets for maximum diversity
  • Seed histories truncated to 5 messages to match the real sliding window constraint
2

Prompt Engineering

prompts/generate_telco_chat.yaml → build_prompt()
  • Jinja2 YAML template system with system + user message structure
  • System prompt encodes Thai telco domain expertise, formatting rules, and constraint specifications
  • User prompt injects: intent name, description, and seed examples as few-shot demonstrations
  • Constraints embedded directly: max 5 history messages, typo distribution ratios, intent switching requirements, history length distribution targets
3

LLM Generation

litellm.completion() → Qwen3-235B-A22B-Instruct
  • Uses litellm abstraction layer — model-agnostic, provider-agnostic
  • Default model: Qwen3-235B-A22B-Instruct on Nebius TokenFactory
  • OpenAI-compatible API, temperature 0.8, max_tokens 16384
  • Sequential processing: one intent at a time, ~25s per intent on cloud inference
4

Parsing & Validation

parse_json_output() → validate_examples()
  • JSON extraction from raw LLM output — handles markdown fences, partial output, recovery
  • Schema validation: ensures user_message, intent, session_history fields present and valid
  • History truncation: programmatic enforcement of max 5 message sliding window
  • Intent correction: forces the correct intent label regardless of what the LLM outputs
5

Output

generated/{INTENT}.json
  • Saves to generated/ directory as individual JSON files per intent
  • Exact same format as seed data — seamless integration with downstream training pipeline
  • Summary report with per-intent counts and total statistics
Data Schema

Training Example Format

Each example captures a user's current message, the target intent, and the conversation context — exactly as the real classifier receives it at inference time.

{ "user_message": "เช็คยอดค้างชำระหน่อยค้าบ 😅", ← current message to classify "intent": "MOBILE_ACCOUNT_CHECK_PAYMENT_STATUS_POSTPAID", ← target label "session_history": [ ← sliding window (max 5 messages) { "role": "user", "content": "สัญญาณเน็ตช้ามากเลยครับ" }, ← previous topic (network) { "role": "assistant", "content": "ขออภัยค่ะ ทีมงานกำลังตรวจสอบ" } ← intent switch happens next ] }
user_message

The current chat message to classify. This is the primary model input. Includes natural Thai with typos, slang, emoji, and occasional English.

intent

One of 10 possible target labels. Determined solely by the current user_message, not by the session history.

session_history

Sliding window of the last 5 messages providing conversation context. May contain topic switches from completely different intents — the classifier must learn to ignore irrelevant history.

Rationale

Key Design Decisions

Every constraint in the pipeline exists because of a real-world failure mode. Click to expand the reasoning.

5
Sliding Window = 5 Messages
▾
The production system passes the last 5 messages of the conversation as context to the classifier. Training data must match inference conditions exactly — if you train on 8-message histories but deploy with 5-message windows, the model learns patterns it will never see in production. The pipeline enforces this both in the prompt (instructing the LLM) and programmatically (truncating any violations to the last 5).
⇄
Intent Switching in History
▾
Real users change topics mid-conversation: "เน็ตช้ามาก... อ้อ แล้วบิลเดือนนี้ต้องจ่ายวันไหนครับ". Without intent-switch examples, the model overfits to session history — if the history mentions roaming, it predicts a roaming intent regardless of the current message. At least 5–6 of every 20 examples contain history about a completely different topic, teaching the model that user_message alone determines intent.
✏
Typos & Informal Language
▾
Real Thai chat is messy: เนต vs เน็ต, ป่าว vs ไหม, doubled vowels ค่าาา, slang 555, missing spaces, emoji. If training data is clean Thai only, the classifier fails on real user input. The pipeline requires 8–10 of 20 examples to contain realistic typos, misspellings, and slang — including specific patterns like ตรจสอบ, รบกวณ, กานใช้งาน.
🌐
Bilingual Examples (Thai + English)
▾
AIS serves expats, tourists, and bilingual Thai users who type in English. Messages like "can u check my bill pls" or "how much roaming data left ah" are real usage patterns. The pipeline includes 2–3 English examples per intent with intentional grammar mistakes reflecting non-native speakers, ensuring the classifier handles both languages.
🎯
Few-Shot Seed Strategy
▾
Seeds aren't randomly selected — the select_seed_examples() function uses round-robin across history-length buckets (0, 2, 4, 5 messages). This ensures the LLM sees the full distribution of conversation depths, not just one pattern. A random 5-sample pick might yield all long conversations, causing the LLM to only generate long histories.
💬
Sentence & Word Diversity
▾
Without explicit instructions, LLMs fall into repetitive phrasing patterns — generating 15 out of 20 user_message values starting with "อยากเช็ค..." or "ขอดู...". Real users express the same intent in wildly different ways: questions ("ยอดเท่าไหร่?"), imperatives ("ช่วยดูยอดให้หน่อย"), fragments ("ยอดค้างชำระ"), or casual conversation ("เอ่อ พอดีจะดูยอดน่ะ"). The prompt now explicitly requires varied verbs, sentence forms, directness levels, and message lengths, with a hard cap of 2–3 examples sharing the same opening word. Each example should feel like it came from a different person.
Why This Pipeline

SDG Pipeline vs. Handcrafted ChatGPT

The traditional approach — manually prompting ChatGPT, copy-pasting outputs, fixing JSON errors — doesn't scale and produces inconsistent results. Here's how the automated pipeline compares.

Legacy Handcrafted ChatGPT

  • Manual copy-paste into ChatGPT, re-prompt each intent individually
  • No version control — each session is ad-hoc, prompts vary by mood
  • Inconsistent quality: different wordings yield different styles and formats
  • No validation — raw output may have wrong JSON, extra fields, broken structure
  • Zero reproducibility — can't re-run to get comparable output
  • Time-consuming: 2–4 hours of manual work for 200 examples
  • No seed strategy — manually picking examples each time
  • No constraint enforcement — history length, typo ratio, intent switching all manual
  • No scalability — need 1000 examples? Repeat the entire process 5x
  • Data leakage risk — ChatGPT may memorize or repeat seeds verbatim
  • Manual reformatting needed to match training pipeline format

Automated SDG Hub Pipeline

  • Single command generates all 200 examples in ~5 minutes end-to-end
  • Version-controlled YAML prompts stored in git — fully auditable
  • Consistent quality: same system prompt, temperature, constraints every run
  • Built-in validation: JSON parsing, schema enforcement, history truncation
  • Fully reproducible: same seed + prompt + model = comparable output
  • Scalable: change --num-examples 100 for 1000 examples instantly
  • Smart seed selection: algorithm picks diverse examples across history lengths
  • Programmatic constraint enforcement: max-5 truncation, distribution targets in prompt
  • Model-agnostic: swap Qwen, Llama, GPT-4o with one --model flag
  • Provider-agnostic: Nebius, OpenAI, Ollama, vLLM via litellm abstraction
  • Output matches training format exactly — zero reformatting needed
  • Retry logic: failed intents re-run individually with --intents
  • Cost-trackable: litellm logs token usage for budget estimation

Guidelines

SDG Best Practices

Lessons learned from building production-grade synthetic data for Thai telco intent classification. Each practice addresses a real failure mode.

01

🎯 Match Training to Inference

Training data must mirror production conditions exactly. If the classifier receives a 5-message sliding window at inference, generate training data with the same constraint. Mismatched context lengths cause silent accuracy drops.

Critical Architecture
  • Enforce max history length in prompt AND code
  • Use same tokenization and truncation strategy
  • Validate every example post-generation
02

🔀 Diversify Seed Selection

Don't pick seeds randomly. Use stratified sampling across conversation depths (0, 1, 2, 3, 4, 5 messages) to ensure the LLM sees the full distribution. Random selection biases toward the most common pattern.

Seeding
  • Round-robin across history-length buckets
  • Limit to 5 seeds to prevent prompt bloat
  • Truncate long seeds to match the window constraint
03

⇄ Force Intent Switching

Require 25–30% of examples to have session history about a completely different topic. Without this, the model overfits to history — predicting roaming just because the history mentions roaming, ignoring the actual current message.

Critical Context
  • 5–6 of every 20 examples must switch topics
  • user_message alone determines the intent
  • History should contain plausible prior conversations
04

✏ Embrace Messy Input

Real Thai chat includes typos (เนต vs เน็ต), slang (555), emoji, doubled vowels (ค่าาา), and missing spaces. Clean-only training data fails on real users. Target 40–50% informal examples.

Language Realism
  • 8–10 of 20 examples with typos/slang
  • Include agent typos too (realistic support staff)
  • Mix formal and informal register
05

🌐 Include Multilingual Examples

AIS serves bilingual users and expats. Include 10–15% English examples with intentional grammar mistakes typical of non-native speakers. Messages like "can u check my bill pls" are real usage patterns.

Coverage
  • 2–3 English examples per 20-example batch
  • Use non-native English patterns, not perfect grammar
  • Include Thai-English code-mixing in some examples
06

🔒 Mask Sensitive Data

Never expose real phone numbers or OTPs in synthetic data. Use consistent masking formats: 091234XXXX for phone numbers and XXXXXX for OTP codes across all generated examples.

Security
  • Phone format: 0XXXXXXXX with partial masking
  • OTP codes always fully masked
  • Embed masking rules in the system prompt
07

🔧 Validate Programmatically

Never trust LLM output blindly. Validate schema, enforce field types, force correct intent labels, and truncate overlong histories in code. The prompt requests constraints; the code enforces them.

Quality Reliability
  • Check required fields: user_message, intent, session_history
  • Override incorrect intent labels automatically
  • Truncate history exceeding max window size
  • Validate role values are only "user" or "assistant"
08

📑 Version Control Prompts

Store prompt templates as YAML files in git, not in code or chat history. This makes prompts auditable, diffable, and reproducible. A prompt change is a code change — review it like one.

Process
  • Use Jinja2 templates with variable injection
  • Separate system prompt from user prompt
  • Track prompt evolution alongside generated data quality
09

🛠 Stay Model-Agnostic

Use an abstraction layer like litellm so you can swap models (Qwen, Llama, GPT-4o) and providers (Nebius, OpenAI, Ollama) without changing pipeline code. Today's best model isn't tomorrow's.

Architecture
  • One --model flag to switch models
  • OpenAI-compatible APIs for provider portability
  • Temperature 0.8 for creative variation with structure
10

📊 Define Clear Intent Boundaries

Each intent description must specify what it IS and what it ISN'T. Distinguish postpaid vs. prepaid, current vs. historical, roaming vs. domestic. Ambiguous intents produce ambiguous data.

Data Quality
  • 2–3 sentence descriptions per intent
  • List related but different intents to clarify boundaries
  • Include example use cases in intent definitions
11

📈 Control Distribution Targets

Specify exact distribution requirements in the prompt: how many examples per history length, typo ratio, intent switches, and language mix. Without explicit targets, the LLM defaults to its most comfortable pattern.

Quality
  • History length: 2 empty, 3 short, 8 medium, 4 long, 3 max
  • Typo ratio: 40–50% of all examples
  • Intent switch: 25–30% of examples
  • English: 10–15% of examples
12

💬 Maximize Sentence & Word Diversity

LLMs tend to repeat the same phrasing pattern. Explicitly require varied verbs (เช็ค, ดู, อยากรู้, ขอ), sentence forms (questions, statements, fragments, imperatives), directness levels, and message lengths. No two examples should feel like the same person.

Critical Diversity
  • Vary opening words — max 2–3 examples with same opener
  • Mix direct ("เช็คบิล") vs indirect ("รบกวนสอบถามยอดหน่อยค่ะ")
  • Mix lengths: single phrase vs full sentence with context
  • Each user_message should feel like a different person
13

⚡ Fail Gracefully, Retry Smart

LLMs sometimes produce malformed output. Parse JSON with fallbacks (strip markdown fences, try line-by-line). Skip bad examples instead of crashing. Allow re-running individual failed intents with --intents.

Reliability
  • Multi-stage JSON recovery: array → line-by-line → skip
  • Per-intent retry without regenerating everything
  • Log warnings for skipped examples, don't silently drop
