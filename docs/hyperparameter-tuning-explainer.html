The Tuning Strategy

We tune in 4 phases, changing one variable at a time (or a tightly coupled pair). Each iteration builds on the best result from the previous one.

The Golden Rule of Hyperparameter Tuning

Change one thing at a time. If you change learning rate AND LoRA rank simultaneously and the model improves, you don't know which change helped. Systematic tuning = isolate variables, measure impact, keep what works.

Phase 1: Learning Rate
Iterations 1-3
The single most impactful hyperparameter. Sweep 3 values to find the sweet spot.
Phase 2: LoRA Architecture
Iterations 4-6
How many parameters to adapt? Which layers? Rank, alpha, and target modules.
Phase 3: Regularization
Iterations 7-8
Fight overfitting: dropout, weight decay, epochs. Critical with only ~228 training samples.
Phase 4: Polish & Combine
Iterations 9-10
Combine winners. Try batch size & scheduler tweaks. Final best model.

Decoder Model: qwen2.5-3b-instruct

A parallel tuning plan for the decoder-based generative model. Five focused cycles, each isolating a tightly coupled pair of hyperparameters.

qwen2.5-3b-instruct + LoRA · Decoder SFT · 5 Cycles
C1

Cycle 1: Learning Rate & Warmup

Aggressive LR with zero warmup — test whether the decoder converges faster without a ramp-up phase.

1

Learning Rate & Warmup

LEARNING RATE
▼
ParameterBaselineThis RunWhy
learning_rate 2e-4 5e-4 LoRA on a 3B decoder has enough parameters to tolerate a higher LR. 2.5× increase to test faster convergence.
warmup_ratio 0.1 0.0 Skip warmup entirely. With cosine scheduling, the LR already starts high and decays naturally. Zero warmup means the model begins learning at full speed from step 1.
Why this combination? For decoder models with LoRA, the warmup phase can waste precious steps on a small dataset. By jumping straight to the higher LR and letting cosine decay handle the annealing, we maximize learning in the limited training budget. If gradient norms stay stable, warmup was unnecessary overhead.
# TrainingArguments changes: learning_rate=5e-4, # was 2e-4 warmup_ratio=0.0, # was 0.1
Exploding Gradients. By removing the gentle warmup and aggressively increasing the speed, you will “fry” the attention layers on Epoch 1. The training loss will plateau or shoot to NaN, and the model will mode-collapse — guessing the same intent every time.
Takeaway: Warmup exists for a reason. Decoder attention layers are sensitive to initial gradient magnitude. Without a ramp-up phase, the 5e-4 LR sends catastrophically large updates through the transformer on step 1, destroying the pretrained representations before learning can begin.
C2

Cycle 2: LoRA Rank & Alpha

Quadruple the adaptation capacity — decoder models may need more expressiveness for generative tasks.

2

LoRA Rank & Alpha

LORA ARCHITECTURE
▼
ParameterBaselineThis RunWhy
r 16 64 4× increase in rank. A 3B model has far more internal dimensions to adapt — higher rank lets LoRA capture richer task-specific representations.
lora_alpha 32 128 Maintain alpha/r = 2 ratio for consistent effective scaling.
Why r=64? Decoder models used for generative SFT need to learn more complex output distributions than a classifier. The baseline r=16 may be too constrained — r=64 gives the adapters 64-dimensional matrices, dramatically increasing the "budget" for task adaptation. The tradeoff: ~4× more trainable parameters, so watch for overfitting.
peft_config = LoraConfig( r=64, # was 16 lora_alpha=128, # was 32 (keep ratio 2x) ... )
Diminishing Returns. You will give the adapter a massive “brain upgrade.” The adapter file size jumps from ~20MB to ~80MB, and VRAM usage spikes noticeably. However, accuracy barely moves from ~95%, proving that text classification doesn’t require high-rank matrices.
Takeaway: More parameters ≠ better performance. For a structured classification task, r=16 already captures enough task-specific signal. The 4× increase in trainable parameters adds cost (memory, storage, inference latency) with no meaningful accuracy gain. Always match model capacity to task complexity.
C3

Cycle 3: Epochs & Dropout

Train for longer with zero LoRA dropout — test whether the model benefits from more passes and full adapter utilization.

3

Epochs & Dropout

REGULARIZATION
▼
ParameterBaselineThis RunWhy
num_train_epochs 5 15 3× more epochs. With higher rank LoRA from Cycle 2, the model has more parameters to converge — it may need more passes through the data to fully utilize them.
lora_dropout 0.05 0.0 Remove dropout entirely. With a 3B decoder using high-rank LoRA, the adapters are already a small fraction of total params. Dropout may be unnecessarily limiting the adapter's learning capacity.
Why zero dropout with more epochs? This is a counterintuitive but valid approach for decoder SFT: the pretrained 3B model already has strong regularization from pretraining, and the LoRA adapters represent a tiny fraction of total parameters. Removing dropout lets every adapter neuron contribute every step, while the extended training (15 epochs) gives the model more time to learn with this full capacity. If overfitting appears, use early stopping from Cycle 3 onwards.
# TrainingArguments: num_train_epochs=15, # was 5 # LoRA config: lora_dropout=0.0, # was 0.05
The U-Curve. By turning off dropout (anti-memorization) and forcing the model to read the data 15 times, you will watch train_loss drop to near zero — but eval_loss will start curving back upward around Epoch 10. The model is memorizing the training data and losing its ability to generalize.
Takeaway: This is the textbook overfitting U-curve in action. Dropout is not optional decoration — it forces the model to learn redundant, robust features instead of brittle memorized patterns. Combined with excessive epochs, you get a model that aces the training set but fails on anything new. The divergence point between train and eval loss is exactly where you should have stopped.
C4

Cycle 4: Batch Sizing

Smaller per-device batch with higher gradient accumulation — more update steps per epoch for smoother learning.

4

Batch Sizing

BATCH ARCHITECTURE
▼
ParameterBaselineThis RunWhy
per_device_train_batch_size 16 4 4× smaller per-device batch. This means each forward pass processes fewer examples, reducing memory pressure and allowing longer sequences if needed.
gradient_accumulation_steps 1 4 Accumulate gradients over 4 micro-batches. Effective batch size = 4 × 4 = 16 (unchanged), but with 4× more update steps per epoch.
Why this configuration? A 3B decoder model is memory-hungry. Reducing per-device batch to 4 frees significant GPU memory, enabling you to potentially increase max_seq_length or avoid OOM issues. Meanwhile, gradient_accumulation_steps=4 preserves the effective batch size at 16, so gradient quality is maintained. The bonus: 4× more optimizer steps per epoch, giving the LR scheduler much finer control.
Effective batch size = per_device_batch × grad_accum = 4 × 4 = 16 (unchanged)
# TrainingArguments: per_device_train_batch_size=4, # was 16 gradient_accumulation_steps=4, # was 1 # effective batch = 4 * 4 = 16 (unchanged)
Out of Memory (OOM). By trying to process 4 texts simultaneously on a single 16GB T4 GPU, the pipeline will fatally crash with a CUDA Out of Memory error. The 3B decoder model’s activations at batch_size=4 exceed what the T4’s VRAM can hold.
Takeaway: gradient_accumulation is the mathematical magic trick used to simulate large batch sizes when hardware is constrained. Instead of fitting 4 samples in one forward pass (OOM), you process 1 sample at a time and accumulate the gradients over 4 steps — mathematically equivalent, but memory-feasible. always calculate your memory budget before changing batch sizes on large models.
C5

Cycle 5: Target Modules

Refine which attention projections receive LoRA adapters for the decoder architecture.

5

Target Modules

LORA ARCHITECTURE
▼
ParameterBaselineThis RunWhy
target_modules default / all linear ["q_proj", "v_proj"] Focus LoRA adapters exclusively on the query and value projections in the attention layers. This is the classic LoRA configuration from the original paper.
Why q_proj + v_proj only? The original LoRA paper (Hu et al., 2021) found that adapting query and value projections captures the most task-relevant attention patterns with the fewest trainable parameters. For Qwen2.5's architecture, q_proj controls "what to attend to" and v_proj controls "what information to extract." Omitting k_proj, o_proj, and MLP layers reduces trainable params significantly, acting as a natural regularizer for small datasets.
peft_config = LoraConfig( target_modules=["q_proj", "v_proj"], # focused on attention Q/V only ... )
Encoders vs. Decoders. By reverting the adapter to only q_proj and v_proj (the standard BERT/encoder configuration), the model will successfully train — but accuracy will plummet back to ~50-60%.
Takeaway: A 3B generative decoder requires LoRA attachments on all linear layers to forcefully override its conversational pre-training. Unlike encoders (which are pretrained for understanding), decoders are pretrained for generation — adapting only 2 of many projection layers isn’t enough to redirect the model’s deeply ingrained autoregressive behavior toward classification. This is the fundamental architectural difference between encoder and decoder fine-tuning.
