First, What is Fine-Tuning?

Before we look at the metrics, let's understand what's actually happening during SFT with an everyday analogy.

The Student Analogy

Imagine a student (phayathaibert) who already speaks Thai fluently (pre-training) but has never worked in telecom customer service.

SFT = giving them a study guide of 285 example customer messages with the correct intent labels, and having them practice over and over (5 epochs = 5 full passes through the study guide).

Each metric we'll cover maps to something about this student's learning journey:

When the student memorizes exact answers instead of learning the concept — fails on new questions

The Building Blocks: Sample, Batch, Step, Epoch

Before we look at the charts, you need to understand these 4 terms. They are the units of measurement for everything that happens during training.

1. Sample = One Training Example

A sample is a single input-output pair the model learns from. In your case, each sample is:

INPUT (customer message):
"How much do I owe on my postpaid bill?"
→
LABEL (correct intent):
PAYMENT_POSTPAID

Your training set has ~285 samples total — 285 customer messages, each labeled with one of the 10 intents.

2. Batch = A Group of Samples Processed Together

The model does not learn one sample at a time. Instead, it groups samples into batches and processes them together. This is much faster (GPUs are designed for parallel computation) and also makes learning more stable.

The batch size is a hyperparameter you choose. Common values are 4, 8, 16, or 32. In your run, the effective batch size is approximately ~7 (we can deduce this from the math: 285 samples / 43 steps per epoch ≈ 6.6, rounded up).

Why batch size matters:
TOO SMALL (1-2)
Noisy updates, slow training, grad norm spikes
GOOD RANGE (4-16)
Balanced stability and speed. Your ~7 fits here.
TOO LARGE (64+)
Needs more GPU memory, may generalize worse

3. Step = One Batch Processed + One Model Update

A step (also called an iteration) is the atomic unit of training. In each step, three things happen:

①
FORWARD PASS
Feed one batch (~7 samples) through the model. Get predictions.
②
COMPUTE LOSS
Compare predictions to correct labels. Calculate how wrong the model was.
③
BACKWARD PASS
Calculate gradients and update model weights. This is where learning happens.

1 step = 1 batch = ~7 samples processed = 1 weight update.
The X-axis on all your training charts says "Step" — now you know what that means.

4. Epoch = One Complete Pass Through ALL Samples

An epoch is completed when the model has seen every training sample exactly once. Since you process ~7 samples per step, it takes ~43 steps to get through all 285 samples = 1 epoch.

loss (training loss) CORE METRIC

How badly is the model getting the training examples wrong? Lower = better. This should decrease smoothly during training.

What loss actually measures

When the model sees the message "How much do I owe on my postpaid bill?" and the correct answer is PAYMENT_POSTPAID, the model outputs a probability for each of the 10 intents.

If it says 90% confident PAYMENT_POSTPAID → low loss (good!)

If it says 10% confident PAYMENT_POSTPAID → high loss (bad!)

The mathematical function used is cross-entropy loss — it measures how far the model's probability distribution is from the correct answer.

training loss journey

This is a healthy training loss curve. It drops steeply at first (the model quickly learns the obvious patterns), then gradually levels off (it's refining the harder cases). Starting at ~3.5 and ending at ~0.65 shows strong learning.

What if the loss curve looked different?

SMOOTH DECREASE ✓
What s looks like. Healthy training. The model is learning steadily.
FLAT / NOT DECREASING
Model isn't learning. Learning rate might be too low, or data has issues.
SPIKY / ERRATIC
Unstable training. Learning rate too high, or batch size too small.
eval_loss CORE METRIC

Same as training loss, but measured on data the model has NEVER seen. This is the honest report card.

Training loss vs eval loss — the critical difference

Training loss = how well the student does on practice questions they've already studied.
Eval loss = how well the student does on a pop quiz with completely new questions.

Training loss will almost always go down (the student memorizes). But if eval loss starts going up while training loss is still going down — that's overfitting.

eval loss journey

The eval loss drops beautifully from 2.5 to 0.65 across epochs 1-4 — but rises slightly at epoch 5 (from 0.65 to ~0.70). This is the first sign of overfitting.

The student analogy for overfitting

Imagine our student read the study guide so many times that they memorized the exact wording of each question-answer pair. When asked "How much do I owe on my postpaid bill?" — they know it perfectly because they've seen that exact sentence 5 times.

But when a NEW customer asks "Can you check my monthly plan charges?" (same intent, different wording) — the student stumbles because they memorized specific words instead of understanding the underlying concept.

This is especially likely with only 285 training samples — the model sees each example ~5 times across 5 epochs, giving it plenty of opportunity to memorize.

How to spot overfitting — the classic pattern

The pattern: training loss keeps dropping (model memorizes better), but eval loss starts rising (performs worse on new data). The gap between them widens. The best checkpoint is right where eval loss was lowest.

learning_rate HYPERPARAMETER

How big of a step does the model take to correct itself after each mistake? Controls the speed of learning.

The driving analogy

Learning rate is like driving to a destination:
High LR (start) = driving fast on the highway — you cover lots of ground quickly, but you might overshoot exit.
Low LR (end) = slow, careful driving in a parking lot — you're close to spot, making tiny adjustments.

learning rate schedule: Linear Decay

run used linear decay: the learning rate starts at 2e-4 (0.0002) and decreases linearly to 0 by the end of training. This is a standard and safe schedule — it prevents the model from making large, destabilizing changes in later epochs.

What does 2e-4 actually mean?

2e-4 = 0.0002 — the model adjusts its weights by about 0.02% in the direction of each correction. This sounds tiny, but across millions of parameters and thousands of steps, these small adjustments add up to major behavior changes.

TOO HIGH (1e-2)
Overshoots, unstable, loss explodes
JUST RIGHT (2e-4) ✓
choice. Standard for BERT fine-tuning
TOO LOW (1e-6)
Barely learns, wastes compute
grad_norm STABILITY INDICATOR

How large are the corrections the model is making at each step? Measures the magnitude of gradients.

What gradients are (simplified)

After the model makes a prediction and checks it against the correct answer, it calculates "how should I adjust each of my millions of parameters to do better next time?" These adjustments are called gradients.

Grad norm = the overall size of all these adjustments combined into a single number. Think of it as "how urgently does the model need to change?"

grad norm: healthy pattern

DECREASING ✓
s. Model needs less correction as it learns.
STABLE / FLAT
OK, but model may not be converging.
EXPLODING ↑↑
Dangerous! Training is diverging. Reduce LR.

Why infra metrics matter

If samples_per_second drops suddenly, GPU might be throttling (overheating) or competing with other processes. If runtime increases between runs with the same data, something changed in environment. These metrics are debugging tools for hardware/speed issues — not model quality.

Key Takeaway: Decoders Trade Speed for Capability

The decoder run used ~75× more compute and took ~53× longer than the encoder. This is the fundamental cost-performance tradeoff — decoder models can handle generative tasks (chat, summarization, reasoning) that encoders cannot, but they require significantly more GPU time, memory, and FLOPs per training step. When choosing between encoder vs decoder SFT, consider whether the task truly needs generation or if a classification head on an encoder is sufficient.

Best Practices for Decision Making

Based on what you see in training run, here are actionable guidelines for this and future fine-tuning runs.

1. Always use the best checkpoint, not the last one

final model (epoch 5) is not best model. Epoch 4 had higher eval accuracy (0.80 vs 0.78) and lower eval loss (0.65 vs 0.70). Most training frameworks like HuggingFace Transformers have a load_best_model_at_end=True option that automatically keeps the best checkpoint.

  • Set evaluation_strategy="epoch" to evaluate after every epoch
  • Set load_best_model_at_end=True to auto-select the best checkpoint
  • Set metric_for_best_model="eval_loss" (or "eval_accuracy")
  • Set save_total_limit=3 to avoid disk bloat from too many checkpoints

2. Use early stopping to prevent overfitting

Instead of guessing the right number of epochs, let the training stop automatically when eval loss stops improving. If you had early stopping with patience=1, training would have stopped after epoch 4 — saving you compute and getting a better model.

# HuggingFace example:
from transformers import EarlyStoppingCallback

trainer = Trainer(
  ...
  callbacks=[EarlyStoppingCallback(early_stopping_patience=2)]
)

3. With only 285 samples, consider data augmentation

285 samples across 10 classes = ~28.5 samples per class on average. That's very few. The model overfits because it doesn't have enough variety. Options to consider:

A Get more labeled data

The most reliable fix. Even going from 285 to 1000 samples can significantly reduce overfitting and improve generalization.

B Paraphrase augmentation

Use an LLM to generate paraphrases of existing samples. "How much do I owe?" → "What's my outstanding balance?" Same intent, different wording.

C Back-translation

Translate Thai → English → Thai. The retranslation naturally paraphrases while keeping the intent. Works well for Thai NLP.

D Reduce epochs

Train for 3-4 epochs instead of 5. With this small dataset, less is more. The sweet spot was epoch 4.

4. What to watch during training — the checklist

Every time you start a fine-tuning run, monitor these signals in order of priority:

PrioritySignalHealthyAlarm
P0 Training loss Smoothly decreasing Flat, erratic, or increasing
P1 Eval loss Decreasing with train loss Starts increasing = overfitting
P1 Eval accuracy Increasing then plateaus Drops after plateau = overfitting
P2 Grad norm Gradually decreasing Exploding upward = unstable training
P3 Learning rate Follows expected schedule Unexpected values = config bug

5. Decision flowchart: What to do next with model

Q: Is eval accuracy (0.74-0.80) good enough for production?

If YES: Deploy the epoch 4 checkpoint. Set up monitoring on real traffic to track accuracy drift over time.

If NO, need higher accuracy:

• First, check per_class_metrics.csv — which intents have the lowest F1? Focus data collection there.

• Get more training data (especially for confused intent pairs).

• Try a larger base model if compute allows.

• Experiment with hyperparameters: lower LR (1e-4), fewer epochs (3-4), weight decay.
