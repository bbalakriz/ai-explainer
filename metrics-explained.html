The Scenario: Mobile Customer Intent Classification

You work at a mobile telecom company. Customers send messages about their postpaid bills, prepaid top-ups, data packages, roaming, and SIM management. You've fine-tuned an ML model to read each message and classify it into one of 10 intent categories so it gets routed to the right support team.

Here are all 10 intents model must choose from:

MOBILE_ACCOUNT_CHECK_PAYMENT_STATUS_POSTPAIDCheck payment status, outstanding balance, or billing for postpaid accounts
MOBILE_ACCOUNT_CHECK_PAYMENT_STATUS_PREPAIDCheck remaining balance, top-up history, or credit expiry for prepaid SIMs
MOBILE_BILLING_CHECK_DUE_DATECheck when the next bill payment is due or billing cycle dates
MOBILE_PACKAGE_CHECK_DATA_HISTORYView past data package subscriptions, usage history, and add-on purchases
MOBILE_ROAMING_CHECK_DATA_CURRENTCheck current roaming data balance while traveling abroad
MOBILE_ROAMING_CHECK_DATA_HISTORYView past roaming data usage across previous trips
MOBILE_ROAMING_CHECK_PAYMENT_POSTPAIDCheck roaming charges and payment status on postpaid bill
MOBILE_ROAMING_SUBSCRIBE_DATA_POSTPAIDSubscribe to a roaming data package for postpaid lines
MOBILE_SIM_CHECK_UNSUBSCRIBE_DATA_POSTPAIDCheck or cancel data packages on postpaid SIM
MOBILE_SIM_CHECK_UNSUBSCRIBE_DATA_PREPAIDCheck or cancel data packages on prepaid SIM

The model was evaluated on 86 real customer messages. The 5 metrics below measure how well it performed. Let's make this concrete with a fixed example using 3 of the above intents so you can see the differences instead of memorizing formulas.

That's it. But here's the problem...

Accuracy doesn't tell you where the model struggles. In our example, 3 errors all involved confusing PAYMENT_POSTPAID with the other two classes. But accuracy just says "70% correct" — it hides which intents are causing trouble.

That's why we need Precision and Recall — they zoom into each class individually.

Here's the key insight most people miss

Accuracy = 0.70 (7 correct out of 10 — one number for everything)

But the per-class story is very different:
• POSTPAID: Precision 0.50 — the model over-predicts this class (it's a "magnet" for errors)
• DUE_DATE: Precision 1.00 but Recall 0.67 — when it says due date it's always right, but it misses some
• PREPAID: Balanced at 0.75/0.75 — the best performing class

Accuracy hides all this detail. Precision, Recall, and F1 reveal it.

Computing the overall (macro-average) scores

Evaluation dashboard reports a single number per metric. That's the average across all classes:

ClassPrecisionRecallF1
PAYMENT_POSTPAID 0.50 0.67 0.57
CHECK_DUE_DATE 1.00 0.67 0.80
PAYMENT_PREPAID 0.75 0.75 0.75
Average (macro) (0.50+1.00+0.75)/3 = 0.75 (0.67+0.67+0.75)/3 = 0.70 (0.57+0.80+0.75)/3 = 0.71

Notice how Accuracy (0.70) and average F1 (0.71) are close but not identical. They measure different things: accuracy treats every message equally, while F1 averages per-class performance (giving each class equal weight regardless of how many messages it has).

recall_score 0.74

Out of all messages that truly belong to a class, how many did the model find?
"Out of all real PREPAID messages, how many did I catch?"

Recall = TP / (TP + FN)

Look at PAYMENT_PREPAID: there were 4 real prepaid messages (IDs 7, 8, 9, 10). The model found 3 of them but missed ID 8 — it thought that top-up complaint was a postpaid issue.

Recall = 3 / (3+1) = 0.75

Why recall matters: Low recall means real cases slip through. If a prepaid customer's balance complaint gets routed to the postpaid team, they get the wrong kind of help.

Recall = TP / (TP + FN)  —  "Of all real X, how many did I catch?"
Model's average recall: 0.74 — it catches 74% of messages per class

Precision vs Recall — The Key Tradeoff

Precision = "When I say X, am I right?" → TP / (TP + FP)
Recall    = "Of all real X, did I catch them?" → TP / (TP + FN)

Model has higher recall (0.74) than precision (0.64). It casts a wide net (catches most real cases) but sometimes labels things with the wrong intent (false alarms).

f1_score 0.68

A single number that balances Precision and Recall. The harmonic mean of both.

The F1 formula

F1 = 2 × (Precision × Recall) / (Precision + Recall)
Same thing using TP, FP, FN:
F1 = 2TP / (2TP + FP + FN)

Why harmonic mean, not simple average?

Imagine a lazy model that always predicts POSTPAID for everything:

  • Precision for POSTPAID = 3/10 = 0.30 (only 3 of 10 predictions are actually postpaid)
  • Recall for POSTPAID = 3/3 = 1.00 (it "found" all 3 postpaid messages... by guessing everything)
  • Simple average = (0.30 + 1.00) / 2 = 0.65 — sounds OK!
  • F1 = 2 × (0.30 × 1.00) / (0.30 + 1.00) = 0.46 — correctly reveals the model is poor

F1 rewards models that do well on BOTH precision AND recall, not just one.

How to read this:
• Diagonal (2, 2, 3) = correct predictions. Sum = 7 → Accuracy = 7/10 = 0.70
• POSTPAID column total = 2+1+1 = 4 predictions. Of those, 2 were right → Precision = 2/4 = 0.50
• POSTPAID row total = 2+0+1 = 3 true cases. Model found 2 → Recall = 2/3 = 0.67
• The off-diagonal red numbers show the POSTPAID column absorbs errors from both other classes — it's the "magnet" for misclassifications.

Squares off the diagonal are mistakes. Clusters of off-diagonal marks reveal which intents confuse the model most — likely similar ones like POSTPAID vs PREPAID, or the different roaming intents.

What confusion matrix reveals

(1) Most squares are on the diagonal — the model gets the majority of intents right.
(2) A few off-diagonal clusters exist — these are pairs of similar intents the model confuses (e.g., postpaid vs prepaid, or different roaming sub-intents).
(3) The color intensity varies — some classes have more test examples than others.

What the Numbers Tell Us

Encoder's Strength: Coverage

The encoder catches more real cases per class (recall 0.74 vs 0.69). It casts a wider net — fewer customer messages slip through unrecognised. It also has higher raw accuracy (0.74 vs 0.69), meaning more total predictions land correctly.

Decoder's Strength: Precision

When the decoder does predict a class, it's right 85% of the time vs the encoder's 64%. That's a massive +21 point gap. It makes far fewer false alarms — when it says "postpaid", you can trust it. This high precision lifts its F1 to 0.75 despite lower recall.

The Tradeoff in Plain English

Encoder is like a customer service agent who routes almost every ticket to some team — most get to the right place (74%), but a fair number end up with the wrong team (precision only 64%). Fewer messages fall through the cracks.

Decoder (Qwen2.5-7B) is like a more cautious agent who only routes when confident — when it does route, it's almost always correct (85%), but it misses more messages that needed routing (recall 69%). Fewer wrong handoffs, more missed ones.

Verdict: Decoder (Qwen2.5-7B) is the stronger model

F1 score is the most reliable single metric for classification because it balances both precision and recall into one number — and the decoder wins here (0.75 vs 0.68).

The encoder's higher accuracy (0.74 vs 0.69) can be misleading. Accuracy treats every prediction equally, but in a multi-class problem with varying class sizes, it can be inflated by getting the common classes right while struggling on rarer ones. F1's macro-average gives every class equal weight, which better reflects real-world routing quality.

The decoder's +21 point precision advantage is the decisive factor. In a customer support routing system, sending a message to the wrong team (false positive) creates a worse experience than a slight delay in routing (missed case) — because the customer has to be re-routed, leading to longer resolution times and frustration. The decoder minimises these costly misroutes.

Bottom line: The decoder (Qwen2.5-7B) should be chosen. It delivers higher F1, dramatically higher precision, and a more trustworthy classification — when it predicts an intent, you can rely on it being correct 85% of the time.
